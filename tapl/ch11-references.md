# References & Pointers

In most programming languages, you have two ways to work with data: you can
_own_ it (a copy lives in your variable) or you can _borrow_ it (your variable
holds an address that points to someone else's data). Auto makes this
distinction explicit with a layered reference system that draws from Rust's
borrowing rules and C's pointer model, but presents them in a cleaner, more
intuitive syntax.

In this chapter, we'll cover:

- **What Are References?** -- Borrowing vs owning
- **`view T`** -- Read-only references (like `&T` in Rust)
- **`mut T`** -- Mutable references (like `&mut T` in Rust)
- **The Reference Rules** -- One mutable OR many immutable
- **`*T`** -- Raw pointers for system-level access
- **`&self` in Methods** -- Self-references in method extensions
- **Lifetimes** -- How long references live

## What Are References?

Every variable in Auto owns its value. When you assign `var x = 5`, the value
`5` belongs to `x`. When `x` goes out of scope, the value is cleaned up. This
is the _ownership_ model you've been using since Chapter 1.

But sometimes you don't want to transfer or copy a value -- you just want to
_look at it_ or _modify it in place_. That's what references are for. A
_reference_ is a handle that lets you access data without taking ownership of
it.

Auto provides three levels of reference access, each with increasing power and
corresponding restrictions:

| Reference | Auto | Rust | C++ | What it does |
|-----------|------|------|-----|--------------|
| Read-only | `view T` | `&T` | `const T&` | Read but not modify |
| Mutable | `mut T` | `&mut T` | `T&` | Read and modify |
| Raw pointer | `*T` | `*T` / `*mut T` | `T*` | Unrestricted system access |

This layered design means you can pick the right tool for the job. Most code
uses `view T` and `mut T`, which are safe and checked at compile time. Only
low-level systems code needs `*T`, and it requires explicit opt-in.

## `view T` -- Read-Only References

A `view T` reference lets a function _read_ data without taking ownership.
The original variable remains valid and usable after the function call. This
is Auto's equivalent of Rust's `&T` or C++'s `const T&`.

<Listing number="11-1" file-name="main.at" caption="Using `view T` to read a string without consuming it">

```auto
fn print_length(s view String) {
    print(s.len())
}

fn main() {
    var name = "Hello, Auto!"
    print_length(name)
    print(name)  // still valid -- view doesn't consume
}
```

```rust
    println!("{}", name);  // still valid -- view doesn't consume
```


The function `print_length` takes `s view String`, meaning it receives a
read-only view of the string. It can call `.len()` to read the string's length,
but it cannot modify the string's contents. After the call returns, `name` is
still valid in `main` -- `view` does not consume the value.

Compare this to what would happen without `view`:

```auto
fn consume(s String) {
    // s is now owned here
}

fn main() {
    var name = "Hello"
    consume(name)
    // print(name)  // ERROR: name was moved
}
```

Without `view`, Auto's implicit move semantics would transfer ownership of
`name` into `consume`, and `name` would no longer be usable afterward. The
`view` keyword explicitly opts out of this behavior.

How `view T` maps across languages:

- **Rust** generates a `&T` reference parameter. The caller passes `&name` to
  borrow the value.
- **Python** passes the string directly -- Python strings are immutable, so
  `const` semantics are automatic.
- **C** generates a `const char*` parameter, enforcing read-only access at the
  type level.
- **TypeScript** passes the string directly -- TypeScript strings are immutable
  by default.

## `mut T` -- Mutable References

A `mut T` reference lets a function _read and modify_ data in place. Changes
made through the reference are visible to the original owner. This is Auto's
equivalent of Rust's `&mut T` or C's pointer parameter.

<Listing number="11-2" file-name="main.at" caption="Using `mut T` to modify a value in place">

```auto
fn increment(n mut int) {
    n = n + 1
}

fn main() {
    var x = 5
    increment(x)
    print(x)  // x is now 6
}
```

```rust
// Auto-generated by a2r transpiler

#[allow(unused_imports)]

fn increment(n: &mut i64) {
    *n = *n + 1;
}

fn main() {
    let mut x = 5;
    increment(&mut x);
    println!("{}", x);  // x is now 6
}
```

```python
# Auto-generated by a2p transpiler

    increment(x)
    print(x[0])  # x is now 6

    main()
```

This prints:

The function `increment` takes `n mut int`, meaning it receives a mutable
reference to an integer. Inside the function, `n = n + 1` modifies the original
value. Back in `main`, `x` now holds `6` because the change was made in place.

Key observations:

- The variable `x` in `main` must be declared with `var` (mutable). You cannot
  pass an immutable `let` variable as a `mut T` reference.
- The compiler ensures that `x` is not used while `increment` holds the mutable
  reference. This prevents data races and dangling references at compile time.

How `mut T` maps across languages:

- **Rust** generates a `&mut T` parameter. The caller passes `&mut x`. The
  function body uses `*n` to dereference and assign.
- **Python** has no native mutable references for primitives. The transpiler
  wraps the value in a list (`[5]`) to simulate pass-by-reference semantics.
- **C** generates a pointer parameter (`long long* n`). The caller passes `&x`.
  This is the closest analog in C.
- **TypeScript** wraps the value in an object (`{ value: 5 }`) to simulate
  pass-by-reference, since TypeScript primitives are passed by value.

## The Reference Rules

Auto enforces the same borrowing rules as Rust at compile time. These rules
prevent data races and undefined behavior without any runtime cost:

1. **You can have any number of `view T` (immutable) references to a value** --
   multiple readers are always safe.

2. **You can have at most one `mut T` (mutable) reference to a value** -- only
   one writer at a time.

3. **Immutable and mutable references must not coexist** -- you cannot read
   and write the same value simultaneously.

<Listing number="11-3" file-name="main.at" caption="The reference rules: one mutable reference at a time">

```auto
fn main() {
    var x = 5
    var r1 mut x = x    // OK -- one mutable reference
    // var r2 mut x = x  // ERROR -- only one mutable ref
    print(r1)
    print(x)
}
```

```rust
// Auto-generated by a2r transpiler

#[allow(unused_imports)]
use auto_lang::a2r_std::*;

fn main() {
    let mut x = 5;
    let r1: &mut i64 = &mut x;    // OK -- one mutable reference
    // let r2: &mut i64 = &mut x;  // ERROR -- only one mutable ref
    println!("{}", r1);
    println!("{}", x);
}
```

```python
# Auto-generated by a2p transpiler

def main():
    x = [5]
    r1 = x    # OK -- one mutable reference
    # r2 = x  # ERROR -- only one mutable ref (not enforced in Python)
    print(r1[0])
    print(x[0])

if __name__ == "__main__":
    main()
```

```c
// Auto-generated by a2c transpiler

#include <stdio.h>

int main(void) {
    long long x = 5;
    long long* r1 = &x;    /* OK -- one mutable reference */
    /* long long* r2 = &x; */  /* ERROR -- only one mutable ref (not enforced in C) */
    printf("%lld\n", *r1);
    printf("%lld\n", x);
    return 0;
}
```

```typescript
// Auto-generated by a2ts transpiler

function main(): void {
    let x = { value: 5 };
    let r1: { value: number } = x;    // OK -- one mutable reference
    // let r2: { value: number } = x;  // ERROR -- only one mutable ref (not enforced in TS)
    console.log(r1.value);
    console.log(x.value);
}

main();
```

</Listing>

This prints:

```
5
5
```

The second line is commented out because it would be a compile-time error in
Auto. The compiler rejects `var r2 mut x = x` because `r1` already holds a
mutable reference to `x`. Having two mutable references to the same data would
create undefined behavior -- both could write at the same time, or one could
invalidate the other.

Note that `print(x)` is allowed alongside `r1`. In Auto, reading the original
value through a non-reference path while a mutable reference exists is allowed
in this simple context, because the compiler can verify that `x` is not
modified through any other path during the read.

These rules are strictly enforced by the Auto compiler. In contrast:

- **Python** does not enforce borrowing rules. Multiple variables can reference
  and mutate the same object freely. This is flexible but can lead to subtle
  bugs.
- **C** does not enforce borrowing rules. Multiple pointers to the same data
  can exist, and nothing prevents simultaneous reads and writes. This is a
  common source of memory safety bugs in C programs.
- **TypeScript** does not enforce borrowing rules for the same reasons as
  Python.

> #### Why These Rules Matter
>
> The "one mutable OR many immutable" rule is the key to Auto's memory safety
> guarantees. It eliminates data races at compile time (no two threads can
> simultaneously mutate the same data through Auto's references) and prevents
> iterator invalidation (you can't modify a collection while iterating over it).
> The compiler does all the checking -- there is zero runtime cost.

## `*T` -- Raw Pointers

For systems programming -- writing operating systems, device drivers, or
FFI bindings to C libraries -- Auto provides raw pointers with the `*T`
syntax. Raw pointers bypass the borrowing rules entirely. They can be null,
they can dangle, and they can alias. With great power comes great
responsibility.

<Listing number="11-4" file-name="main.at" caption="Using `*T` raw pointers for direct memory access">

```auto
fn main() {
    var x = 42
    var p *int = &x
    print(*p)    // 42
    *p = 100
    print(x)     // 100
}
```

```rust
// Auto-generated by a2r transpiler

#[allow(unused_imports)]
use auto_lang::a2r_std::*;

fn main() {
    let mut x: i64 = 42;
    let p: *mut i64 = &mut x;
    println!("{}", unsafe { *p });    // 42
    unsafe { *p = 100; }
    println!("{}", x);     // 100
}
```

```python
# Auto-generated by a2p transpiler

def main():
    x = [42]
    p = x
    print(p[0])    # 42
    p[0] = 100
    print(x[0])     # 100

if __name__ == "__main__":
    main()
```

```c
// Auto-generated by a2c transpiler

#include <stdio.h>

int main(void) {
    long long x = 42;
    long long* p = &x;
    printf("%lld\n", *p);    /* 42 */
    *p = 100;
    printf("%lld\n", x);     /* 100 */
    return 0;
}
```

```typescript
// Auto-generated by a2ts transpiler

function main(): void {
    let x: { value: number } = { value: 42 };
    let p: { value: number } = x;
    console.log(p.value);    // 42
    p.value = 100;
    console.log(x.value);     // 100
}

main();
```

</Listing>

This prints:

```
42
100
```

`var p *int = &x` creates a raw pointer `p` that holds the address of `x`.
The `*p` syntax dereferences the pointer to read or write the value it points
to. After `*p = 100`, the original `x` is also `100` because `p` points
directly to `x`'s memory location.

Raw pointers in Auto work identically to C pointers:

- `&x` takes the address of `x`
- `*p` dereferences the pointer
- `*p = value` writes through the pointer

In Rust, raw pointer operations require `unsafe` blocks. The Auto compiler
generates the appropriate safety wrappers. Raw pointers are intended for
advanced use cases:

- Interfacing with C libraries through FFI
- Implementing custom data structures that need pointer arithmetic
- Writing operating system kernels or embedded firmware

For application-level code, prefer `view T` and `mut T` -- they give you the
same performance with compile-time safety guarantees.

## `&self` in Methods

When you define methods on a type using `ext`, you often need the method to
operate on the instance it's called on. Auto uses `&self` in the method
signature to indicate that the method borrows the instance. For methods that
modify the instance, you can use `&mut self`.

<Listing number="11-5" file-name="main.at" caption="Using `&self` in method extensions">

```auto
type Counter {
    count int
}

ext Counter {
    fn increment(&self) {
        self.count = self.count + 1
    }
    fn get(&self) int {
        self.count
    }
}

fn main() {
    var c = Counter { count: 0 }
    c.increment()
    c.increment()
    c.increment()
    print(c.get())  // 3
}
```

```rust

    c.increment();
    c.increment();
    c.increment();
```

The `Counter` type has two methods defined in its `ext` block:

- `increment(&self)` modifies the counter. The `&self` parameter borrows the
  instance, allowing `self.count` to be updated. In Rust, this maps to
  `&mut self` because the method mutates data.
- `get(&self) int` reads the counter's value. The `&self` parameter borrows
  the instance immutably. In Rust, this maps to `&self`.

The `&self` syntax in Auto method signatures maps to the calling convention
you'd expect from object-oriented languages:

- **Rust** generates `&self` or `&mut self` depending on whether the method
  modifies the instance.
- **Python** uses the standard `self` parameter -- Python's class system is
  reference-based by default.
- **C** generates a function with an explicit pointer parameter (`Counter* self`
  or `const Counter* self`). The `->` operator accesses fields through the
  pointer.
- **TypeScript** generates a standard class method with `this` -- TypeScript
  objects are reference types by default.

## Lifetimes

References in Auto do not live forever. Every reference has a _lifetime_ --
the span of code during which the reference is valid. The compiler uses
lifetimes to ensure that no reference outlives the data it points to.

In most cases, Auto can infer lifetimes automatically. You don't need to
write lifetime annotations for simple cases. The compiler analyzes the code
and figures out how long each reference needs to be valid.

<Listing number="11-6" file-name="main.at" caption="References with lifetimes -- the compiler infers validity">

```auto
fn longest(a view String, b view String) view String {
    if a.len() > b.len() {
        a
    } else {
        b
    }
}

fn main() {
    var s1 = "hello"
    var s2 = "world!"
    var result = longest(s1, s2)
    print(result)
}
```

```rust
    let s1 = String::from("hello");
```

This prints:


The function `longest` takes two `view String` references and returns a
`view String` reference pointing to whichever string is longer. The return
value is a reference to one of the inputs -- it doesn't create a new string.

The compiler needs to verify that the returned reference is valid. In this
case, the returned reference points to either `s1` or `s2`, both of which
are still alive when `result` is used in `main`. So the code is valid.

In Rust, this function requires explicit lifetime annotations (`'a`). The
annotation tells the Rust compiler that the returned reference lives as long
as the shorter of the two input lifetimes. Auto's compiler performs the same
analysis but doesn't require you to write the annotations in simple cases like
this.

The compiler would reject code that tries to return a reference to data that
is about to be destroyed:

```auto
fn bad() view String {
    var s = "temporary"
    s  // ERROR: s is about to go out of scope
}
```

This would fail because `s` is a local variable. When `bad` returns, `s` is
destroyed, and the returned reference would point to invalid memory. The
compiler catches this at compile time.

How lifetimes map across languages:

- **Rust** requires explicit lifetime annotations (`'a`) in function signatures
  when the compiler cannot infer them. This is the most explicit system.
- **Python** uses reference counting. Objects are freed when their reference
  count drops to zero, so dangling references are not possible in normal code.
- **C** has no lifetime checking. Returning a pointer to a local variable is
  _undefined behavior_ -- a common source of crashes and security
  vulnerabilities.
- **TypeScript** relies on garbage collection. Like Python, dangling references
  are not a concern because objects are collected when no longer reachable.

## Summary

Auto's reference system provides three levels of access, from safe and
ergonomic to powerful and low-level:

| Concept | Auto | Rust | C | Safety |
|---------|------|------|---|--------|
| Read-only reference | `view T` | `&T` | `const T*` | Compile-time checked |
| Mutable reference | `mut T` | `&mut T` | `T*` | Compile-time checked |
| Raw pointer | `*T` | `*mut T` | `T*` | Programmer's responsibility |
| Method self-borrow | `&self` | `&self` / `&mut self` | `Type* self` | Compile-time checked |
| Lifetime | Inferred | `'a` annotations | None | Compile-time checked |

Key design principles:

1. **Safe by default**: `view T` and `mut T` are checked at compile time. The
   compiler prevents data races, dangling references, and use-after-free bugs
   without any runtime cost.

2. **Explicit opt-in for power**: Raw pointers (`*T`) are available when you
   need them for FFI or systems programming, but you must explicitly choose to
   use them. The compiler won't let you accidentally bypass the safety rules.

3. **Minimal annotation burden**: In most cases, Auto infers lifetimes
   automatically. You only need to think about lifetimes when returning
   references from functions -- and even then, the compiler guides you with
   clear error messages.

4. **Zero-cost abstraction**: References have no runtime overhead. A `view T`
   is the same size as a pointer, and all checking happens at compile time.
   The generated code is as efficient as hand-written C.

`Set` -- which combine ownership and references to provide safe, efficient
dynamic data structures.
