# Collections & Nodes

So far, every variable we've created has held a single value — one number, one
string, one boolean. Real programs deal with _groups_ of things: a list of
names, a table of scores, a set of unique IDs. Auto provides several collection
types that let you store, query, and transform multiple values. In this
chapter, we'll also meet `node`, Auto's keyword for building linked structures
such as chains and trees.

In this chapter, we'll cover:

- **Arrays** — ordered sequences created with `[1, 2, 3]` literal syntax
- **Objects** — named-field records created with `{ name: "Alice", age: 30 }`
- **HashMaps** — key-value maps inferred from string-keyed literals
- **Sets** — deduplicated collections created with `{1, 2, 3}`
- **List operations** — `push`, `pop`, `contains`, `sort`, and more
- **Strings as collections** — `len`, indexing, `split`, `to_upper`
- **Nodes** — linked structures with `node`, `?Item`, and `none`

## Arrays

An _array_ is an ordered, indexed collection of values. You create one with
square brackets and a comma-separated list of elements:

```auto
var numbers = [1, 2, 3, 4, 5]
```

The type is inferred from the elements — since every value here is an `int`,
`numbers` has type `[int]`. You access elements by zero-based index with
`numbers[0]`, and get the length with `numbers.len()`. You can iterate over an
array with `for item in array`.

<Listing number="4-1" file-name="main.at" caption="Creating arrays, indexing, and iterating">

```auto
fn main() {
    var numbers = [1, 2, 3, 4, 5]
    print(numbers[0])
    print(numbers.len())
    var names = ["Alice", "Bob", "Charlie"]
    for name in names {
        print(name)
    }
}
```

Notice how the transpiler adapts the array concept to each language:

## Objects

An _object_ is a collection of named fields. You create one with curly braces
and `key: value` pairs:

```auto
var person = { name: "Alice", age: 30 }
```

The type is inferred from the fields — `person` has fields `name` (type `str`)
and `age` (type `int`). You access fields with dot notation (`person.name`) and
assign new values with dot notation (`person.age = 31`). Because we declared
`person` with `var`, the object is mutable.

<Listing number="4-2" file-name="main.at" caption="Creating an object, reading and writing fields">

```auto
fn main() {
    var person = { name: "Alice", age: 30 }
```

Each language handles object-like data differently:

## HashMaps

When an object literal uses string keys, Auto infers a _HashMap_ instead of a
plain object:

```auto
var scores = { "Alice": 90, "Bob": 85, "Charlie": 92 }
```

The difference from a plain object is that you access values with bracket
notation (`scores["Alice"]`) and the map can grow — you can insert new keys at
runtime. The type is inferred as `Map<str, int>`.

```auto

    var scores = { "Alice": 90, "Bob": 85, "Charlie": 92 }

    print(scores.len())
```

```c
#include <string.h>

typedef struct MapEntry {
    char key[256];
    long long value;
    struct MapEntry* next;
} MapEntry;

typedef struct {
    MapEntry* head;
    int size;
} HashMap;

void map_insert(HashMap* map, const char* key, long long value) {
    MapEntry* entry = map->head;
    while (entry != NULL) {
        if (strcmp(entry->key, key) == 0) {
            entry->value = value;
            return;
        }
        entry = entry->next;
    }
    MapEntry* new_entry = (MapEntry*)malloc(sizeof(MapEntry));
    strcpy(new_entry->key, key);
    new_entry->value = value;
    new_entry->next = map->head;
    map->head = new_entry;
    map->size++;
}

long long map_get(HashMap* map, const char* key) {
    MapEntry* entry = map->head;
    while (entry != NULL) {
        if (strcmp(entry->key, key) == 0) {
            return entry->value;
        }
        entry = entry->next;
    }
    return 0;
}

int main(void) {
    HashMap scores = {NULL, 0};
    map_insert(&scores, "Alice", 90);
    map_insert(&scores, "Bob", 85);
    map_insert(&scores, "Charlie", 92);
    printf("%lld\n", map_get(&scores, "Alice"));
    map_insert(&scores, "Dave", 88);
    printf("%d\n", scores.size);
    return 0;
}
```

The key difference between objects and HashMaps in Auto is the _key type_.
Objects use identifier keys (`{ name: "Alice" }`) and are accessed with dot
notation. HashMaps use string keys (`{ "Alice": 90 }`) and are accessed with
bracket notation. The compiler figures out which one you mean from the syntax.

Look at how each language handles this:

- **Rust** uses `std::collections::HashMap<String, i64>` with `.insert()` and
  `.get()`. The `.unwrap()` on `.get()` is safe here because we know the key
  exists.
- **Python** uses a plain `dict` — the same syntax as Auto. Bracket access and
  `len()` work naturally.
- **C** has no built-in hash map, so the transpiler generates a linked-list-based
  map with `map_insert` and `map_get` helper functions.
- **TypeScript** uses an index signature `{ [key: string]: number }` and
  `Object.keys()` for the length.

## Sets

A _set_ is a collection of unique values. You create one with curly braces and
comma-separated values — the same syntax as an object, but with non-string
elements:

```auto
var unique = {1, 2, 3, 2, 1}
```

Duplicate values are automatically discarded. The compiler infers a `Set<int>`
because the elements are integers. You can add elements with `.add()` and check
membership with `.contains()`.

```auto

    print(unique.contains(3))
}
```

Notice the language differences:

- **Rust** uses `HashSet<i64>` from `std::collections`. The `.insert()` method
  silently ignores duplicates, and `.contains()` takes a reference.
- **Python** uses a native `set` literal. The `in` operator replaces Auto's
  `.contains()` method.
- **C** generates a linked-list-based set with helper functions. Duplicates are
  checked before insertion.
- **TypeScript** uses `Set<number>`. The `.size` property replaces `.len()`, and
  `.has()` replaces `.contains()`.

How does Auto tell the difference between a set `{1, 2, 3}`, an object `{ name:
"Alice" }`, and a HashMap `{ "Alice": 90 }`? By the _keys_:

- **Identifier keys** (no quotes): `{ name: "Alice" }` is an **object**.
- **String keys** (quoted): `{ "Alice": 90 }` is a **HashMap**.
- **No keys** (just values): `{1, 2, 3}` is a **Set**.

The compiler uses these syntactic cues to infer the correct type. You never
need to write `HashMap<String, i64>` or `HashSet<int>` explicitly — Auto
figures it out.

```c
#include <stdlib.h>
#include <stdbool.h>

typedef struct {
    long long* data;
    int size;
    int capacity;
} Vec;

void vec_push(Vec* v, long long value) {
    if (v->size >= v->capacity) {
        v->capacity = v->capacity == 0 ? 4 : v->capacity * 2;
        v->data = (long long*)realloc(v->data, v->capacity * sizeof(long long));
    }
    v->data[v->size++] = value;
}

long long vec_pop(Vec* v) {
    return v->data[--v->size];
}

bool vec_contains(Vec* v, long long value) {
    for (int i = 0; i < v->size; i++) {
        if (v->data[i] == value) return true;
    }
    return false;
}

int compare(const void* a, const void* b) {
    long long diff = *(long long*)a - *(long long*)b;
    return (diff > 0) - (diff < 0);
}

void vec_print(Vec* v) {
    printf("[");
    for (int i = 0; i < v->size; i++) {
        if (i > 0) printf(", ");
        printf("%lld", v->data[i]);
    }
    printf("]\n");
}
```

## Nodes

Arrays, objects, and maps are all _contiguous_ data structures — their elements
are stored next to each other in memory. Sometimes you need a _linked_ structure,
where each element points to the next one. Think of a chain: each link connects
to the next link, and the last link connects to nothing.

Auto provides the `node` keyword for this pattern. A node is like a lightweight
struct where one or more fields can be _optional_ — they might point to another
node, or they might be `none`.

### Defining a Node

You define a node with the `node` keyword, field names, and types. The `?` prefix
on a type means "optional" — the value can either be a real instance or `none`:

```auto
node Item {
    value int
    next ?Item
}
```

This defines a node called `Item` with two fields: `value` (an integer) and
`next` (either another `Item` or `none`). The `?Item` type is Auto's way of
saying "nullable Item" — similar to `Option<Item>` in Rust or `Item | null` in
TypeScript.

<Listing number="4-7" file-name="main.at" caption="Defining a node type and building a linked chain">

```auto
node Item {
    value int
    next ?Item
}

fn main() {
    var third = Item { value: 30, next: none }
    var second = Item { value: 20, next: third }
    var first = Item { value: 10, next: second }
    print(first.value)
    print(first.next.value)
}
```

We build the chain from back to front. First `third` (value 30, next is `none`),

The transpiler maps `node` to each language's idiomatic linked structure:

- **Rust** uses a `struct` with `Option<Rc<RefCell<Item>>>` for the `next`
  field. `Rc` (reference counting) allows multiple ownership, and `RefCell`
  provides interior mutability. This is the standard Rust pattern for linked
  structures.
- **Python** uses a `class` with `__init__`. The `next` field defaults to
  `None`. Python's dynamic typing makes this natural.
- **C** uses a `typedef struct` with a pointer `struct Item* next`. The
  `item_new()` helper function allocates memory with `malloc()`. `NULL` replaces
  `none`.
- **TypeScript** uses a `class` with `next: Item | null`. The non-null assertion
  `first.next!.value` tells TypeScript the value is not null at that point.

### Traversing a Node Chain

Accessing `first.next.value` works for two levels, but real linked structures
can be arbitrarily long. The standard pattern is to start at the head and walk
through each node until you reach `none`:

<Listing number="4-8" file-name="main.at" caption="Traversing a linked chain with a conditional loop">

```auto
node Item {
    value int
    next ?Item
}

fn main() {
    var third = Item { value: 30, next: none }
    var second = Item { value: 20, next: third }
    var first = Item { value: 10, next: second }
    var current = first
    for current != none {
        print(current.value)
        current = current.next
    }
}
```

The traversal pattern is the same in every language:

1. Start with `current = first` (the head of the chain).
2. Loop while `current` is not `none` / `NULL` / `null`.
3. Print `current.value`.
4. Move to the next node: `current = current.next`.

In Auto, `for current != none { ... }` is a conditional loop — it keeps running
as long as the condition is true. The transpiler converts this to `while` in
every target language, since `for cond` is Auto's way of writing what other
languages call `while`.

Notice that the traversal code is nearly identical across all five languages.
Linked list traversal is one of those patterns where the _logic_ is universal —
only the syntax for "is this null?" changes:

| Language | Null check | Null value |
|----------|-----------|------------|
| Auto | `current != none` | `none` |
| Rust | `current.is_some()` | `None` |
| Python | `current is not None` | `None` |
| C | `current != NULL` | `NULL` |
| TypeScript | `current !== null` | `null` |

## Summary

In this chapter, you learned how to work with collections and linked structures
in Auto:

- **Arrays** `[1, 2, 3]` store ordered sequences. Access by index with
  `arr[0]`, get length with `arr.len()`, iterate with `for item in arr`.
- **Objects** `{ name: "Alice" }` store named fields. Access with dot notation
  (`obj.name`), mutate with assignment (`obj.age = 31`).
- **HashMaps** `{ "Alice": 90 }` are inferred from string-keyed object literals.
  Access with brackets (`map["Alice"]`), insert with bracket assignment.
- **Sets** `{1, 2, 3}` store unique values. Add with `.add()`, check membership
  with `.contains()`.
- **List operations** — `.push()`, `.pop()`, `.contains()`, `.sort()` — let you
  manipulate arrays in place.
- **Strings as collections** — `.len()`, `s[0]`, `.split()`, `.to_upper()` —
  treat strings as sequences of characters.
- **Nodes** with `node` and `?T` let you build linked structures. `none` marks
  the end of a chain. Traversal uses `for current != none { ... }`.

The compiler infers collection types from syntax — you never need to write
`Vec<i64>`, `HashMap<String, i64>`, or `HashSet<i64>`. Auto reads your intent
from the literal and picks the right type. In the next chapter, we'll look at
how to define your own custom types with `type` and build methods with `ext`.
