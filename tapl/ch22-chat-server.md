## Design Overview

Before writing any code, let us think about the architecture. A chat server
has three main components:

1. **The server actor** -- a single actor that maintains the global state:
   which users are connected, which rooms exist, and who is in each room. All
   messages flow through this actor.

2. **Client handlers** -- one actor per connected user. Each handler reads
   input from its user, parses commands, and forwards messages to the server
   actor.

3. **The message protocol** -- an enum that defines every type of message
   the system handles: connect, disconnect, chat, private message, join room,
   leave room, room message, and system notifications.

This design follows the actor model from [Chapter 15][ch15]. The server is a
single point of coordination. Client handlers are independent actors that
communicate with the server through messages. No shared memory, no locks, no
data races -- just actors sending messages to each other.

## Step 1: Message Types

Every network protocol starts with a definition of the messages that flow
through it. In Auto, we model this as an `enum`. Each variant represents a
different kind of event in the chat system.

Let us walk through the design decisions:

1. **`Connect(String)`** -- carries the username. When a user connects, the
   server registers them.

2. **`Disconnect(String)`** -- carries the username. The server removes them
   from all rooms and notifies others.

3. **`Chat(String, String)`** -- carries the sender's name and the message
   text. This is a public message broadcast to all users.

4. **`PrivateMsg(String, String, String)`** -- carries the sender, the
   recipient, and the text. Only the recipient should see this.

5. **`System(String)`** -- a notification from the server itself: "Alice
   joined", "Bob left", etc.

Why an enum? Because the compiler forces us to handle every variant. If we
add a new message type later (say, `Kick(String, String)`), the compiler will
warn us at every `is` expression that does not handle it. This is the exhaustive
pattern matching we learned about in [Chapter 7][ch07].

In C, this maps to a tagged union with a manual `switch` statement. In Python,
it maps to a union of dataclasses with `isinstance` checks. In TypeScript, it
maps to a discriminated union with `switch` on the `kind` field. In all cases,
the pattern is the same: one type, many variants, exhaustive handling.

## Step 2: Server Actor

The server actor is the heart of the chat system. It runs as a single actor
(Remember actors from [Chapter 15][ch15]?) that maintains two pieces of
state: a map of connected clients and a message loop.

<Listing number="22-2" file-name="main.at" caption="The server actor managing connected clients">

```auto
// Listing 22-02: Server actor that manages connected clients

enum Message {
    Connect(String),
    Disconnect(String),
    Chat(String, String),
    PrivateMsg(String, String, String),
    System(String)
}

fn server() {
    var clients: Map<String, int> = {}
    for true {
        on msg Message {
            is msg {
                Connect(name) => {
                    clients[name] = 1
                    println("[SYS] " + name + " joined. Online: " + clients.len())
                },
                Disconnect(name) => {
                    clients.remove(name)
                    println("[SYS] " + name + " left. Online: " + clients.len())
                },
                Chat(from, text) => {
                    println(from + ": " + text)
                },
                PrivateMsg(from, to, text) => {
                    println("[PM] " + from + " -> " + to + ": " + text)
                },
                System(text) => {
                    println("[SYS] " + text)
                }
            }
        }
    }
}

fn main() {
    spawn server
    send server Message.Connect("Alice".to_string())
    send server Message.Connect("Bob".to_string())
    send server Message.Chat("Alice".to_string(), "Hello Bob!".to_string())
    send server Message.PrivateMsg("Bob".to_string(), "Alice".to_string(), "Hi Alice!".to_string())
    send server Message.Disconnect("Bob".to_string())
}
```

```python
        try:
            msg = q.get(timeout=1.0)

            elif msg.kind == MessageKind.Disconnect:

            elif msg.kind == MessageKind.Chat:

            break
```

```c
typedef enum MessageKind {
    MsgKind_Connect,
    MsgKind_Disconnect,
    MsgKind_Chat,
    MsgKind_PrivateMsg,
    MsgKind_System,
} MessageKind;

typedef struct Message {
    MessageKind kind;
    const char* field1;
    const char* field2;
    const char* field3;
} Message;

typedef struct {
    char names[MAX_CLIENTS][64];
    int count;
} ClientList;

ClientList clients = { .count = 0 };

void server_handle(Message* msg) {
    switch (msg->kind) {
        case MsgKind_Connect: {
            strncpy(clients.names[clients.count], msg->field1, 63);
            clients.names[clients.count][63] = '\0';
            clients.count++;
            printf("[SYS] %s joined. Online: %d\n", msg->field1, clients.count);
            break;
        }
        case MsgKind_Disconnect: {
            int found = -1;
            for (int i = 0; i < clients.count; i++) {
                if (strcmp(clients.names[i], msg->field1) == 0) {
                    found = i;
                    break;
                }
            }
            if (found >= 0) {
                for (int i = found; i < clients.count - 1; i++) {
                    strncpy(clients.names[i], clients.names[i + 1], 63);
                }
                clients.count--;
            }
            printf("[SYS] %s left. Online: %d\n", msg->field1, clients.count);
            break;
        }
        case MsgKind_Chat:
            printf("%s: %s\n", msg->field1, msg->field2);
            break;
        case MsgKind_PrivateMsg:
            printf("[PM] %s -> %s: %s\n", msg->field1, msg->field2, msg->field3);
            break;
        case MsgKind_System:
            printf("[SYS] %s\n", msg->field1);
            break;
    }
}
```

The server actor maintains a `clients` map that tracks who is online. Each
variant in the `is` expression handles a different message type:

- **`Connect`** adds the user to the `clients` map and announces their arrival.
- **`Disconnect`** removes them and announces their departure.
- **`Chat`** prints the message (in a real server, it would broadcast to all
  clients).
- **`PrivateMsg`** prints a private message indicator.
- **`System`** prints a server notification.

The key insight is that all state changes happen inside a single actor. The
`clients` map is private to the `server` function -- no other actor can touch
it. This eliminates data races without any locks or mutexes. The actor model
guarantees that only one message is processed at a time.

In Rust, the server runs in a spawned thread with an `mpsc::Receiver`. In
Python, it runs in a daemon thread with a `queue.Queue`. In C, we simulate the
message loop with an array of messages. In TypeScript, we process messages
synchronously in an array. The Auto version is the cleanest: `spawn server`,
`send server Message.Connect(...)`. Two lines.

## Step 3: Client Handler

Each connected user needs their own actor that reads input and forwards

```auto
// Listing 22-03: Per-client actor that reads input and forwards messages

enum Message {
    Connect(String),
    Disconnect(String),
    Chat(String, String),
    PrivateMsg(String, String, String),
    System(String)
}

fn client_handler(name String, server_addr int) {
    println("Welcome, " + name + "! Type /quit to disconnect.")
    send server_addr Message.Connect(name.to_string())
    for true {
        on input String {
            if input == "/quit" {
                send server_addr Message.Disconnect(name.to_string())
                println("Goodbye, " + name + "!")
                return
            } else if input.starts_with("/msg") {
                var parts = input.split(" ")
                if parts.len() >= 3 {
                    var target = parts[1]
                    var text = parts[2..].join(" ")
                    send server_addr Message.PrivateMsg(name.to_string(), target.to_string(), text)
                } else {
                    println("Usage: /msg <name> <message>")
                }
            } else if input.len() > 0 {
                send server_addr Message.Chat(name.to_string(), input)
            }
        }
    }
}

fn main() {
    spawn client_handler("Alice".to_string(), 0)
    spawn client_handler("Bob".to_string(), 0)
}
```

```auto
enum Message {
    Connect(String),
    Disconnect(String),
    Chat(String, String),
    PrivateMsg(String, String, String),
    System(String),
    JoinRoom(String, String),
    LeaveRoom(String, String),

    var clients: Map<String, int> = {}
    var rooms: Map<String, List<String>> = {
        "general": [],
        "random": []
    }
    for true {
        on msg Message {
            is msg {
                Connect(name) => {
                    clients[name] = 1
                    rooms["general"].push(name.to_string())
                    println("[SYS] " + name + " joined. Online: " + clients.len())
                },
                Disconnect(name) => {

                        room.value.remove(name)
                    }
                    println("[SYS] " + name + " left. Online: " + clients.len())
                },
                JoinRoom(name, room_name) => {

                        if !rooms[room_name].contains(name) {
                            rooms[room_name].push(name.to_string())
                            println("[SYS] " + name + " joined #" + room_name)
                        }
                    } else {
                        println("[SYS] Room #" + room_name + " does not exist")
                    }
                },
                LeaveRoom(name, room_name) => {
                    rooms[room_name].remove(name)
                    println("[SYS] " + name + " left #" + room_name)
                },
                RoomMsg(room_name, from, text) => {
                    var members = rooms[room_name]
                    println("[#" + room_name + "] " + from + ": " + text)
```

- **`JoinRoom(String, String)`** -- the first field is the username, the second
  is the room name. The server checks that the room exists and that the user
  is not already in it before adding them.

- **`LeaveRoom(String, String)`** -- removes the user from the room's member
  list.

- **`RoomMsg(String, String, String)`** -- a message targeted at a specific
  room. The first field is the room name, the second is the sender, and the
  third is the text. The server looks up the room's member list and would
  deliver the message to each member.

The `rooms` map uses `Map<String, List<String>>` -- a map from room names to
lists of usernames. This is a nested collection, the kind we worked with in
[Chapter 4][ch04] and [Chapter 21][ch21]. Each room name maps to a list of the
users currently in that room.

Notice how the `Disconnect` handler now iterates over all rooms to remove the
disconnected user from every room they were in. This is the `for room in rooms`
syntax from the iterators chapter. It visits every key-value pair in the map.

In C, this nested data structure requires arrays of arrays with manual size
tracking. In Rust, it is `HashMap<String, Vec<String>>`. In Python, a dict of
lists. In TypeScript, a `Map<string, string[]>`. Auto's version is the most
readable: `Map<String, List<String>>`.

```python
    print(f"[#{room_name}] delivered to {len(members)}")

def server(in_q, out_q):
    clients = {}
    rooms = {"general": [], "random": []}
    while True:
        try:
            msg = in_q.get(timeout=1.0)
            if msg.kind == MessageKind.Connect:
                name = msg.field1
                if name in clients:
                    out_q.put(Message(kind=MessageKind.System, field1=f"Name already taken: {name}"))
                else:
                    clients[name] = 1
                    rooms["general"].append(name)
                    out_q.put(Message(kind=MessageKind.System, field1=f"Welcome, {name}!"))
                    out_q.put(Message(kind=MessageKind.System, field1=f"{name} joined. Online: {len(clients)}"))
            elif msg.kind == MessageKind.Disconnect:
                name = msg.field1
                clients.pop(name, None)
                for members in rooms.values():
                    if name in members:
                        members.remove(name)
                out_q.put(Message(kind=MessageKind.System, field1=f"{name} left. Online: {len(clients)}"))
            elif msg.kind == MessageKind.Chat:
                broadcast(clients, rooms, msg)
            elif msg.kind == MessageKind.PrivateMsg:
                if msg.field2 in clients:
                    print(f"[PM] {msg.field1} -> {msg.field2}: {msg.field3}")
                else:
                    print(f"[SYS] User {msg.field2} not found")
            elif msg.kind == MessageKind.JoinRoom:
                name, room_name = msg.field1, msg.field2
                if room_name in rooms:
                    rooms[room_name].append(name)
                    broadcast_room(rooms, room_name, Message(kind=MessageKind.System, field1=f"{name} joined #{room_name}"))
            elif msg.kind == MessageKind.LeaveRoom:
                name, room_name = msg.field1, msg.field2
                if name in rooms.get(room_name, []):
                    rooms[room_name].remove(name)
                broadcast_room(rooms, room_name, Message(kind=MessageKind.System, field1=f"{name} left #{room_name}"))
            elif msg.kind == MessageKind.RoomMsg:
                broadcast_room(rooms, msg.field1, msg)
            elif msg.kind == MessageKind.System:
                print(f"[SYS] {msg.field1}")
        except queue.Empty:
            break

def client_handler(name, server_q):
    print(f"Welcome, {name}! Commands: /msg, /join, /leave, /rooms, /quit")
    inputs = ["Hello everyone!", "/join random", "/rooms", "/quit"]
    for inp in inputs:
        if inp == "/quit":
            print(f"Goodbye, {name}!")
            return
        elif inp.startswith("/msg "):
            parts = inp.split(" ")
            if len(parts) >= 3:
                target = parts[1]
                text = " ".join(parts[2:])
                print(f"[PM] {name} -> {target}: {text}")
        elif inp.startswith("/join "):
            room_name = inp.split(" ")[1]
            print(f"[SYS] {name} joined #{room_name}")
        elif inp.startswith("/leave "):
            room_name = inp.split(" ")[1]
            print(f"[SYS] {name} left #{room_name}")
        elif inp == "/rooms":
            print("[SYS] Available rooms: general, random")
        elif len(inp) > 0:
            print(f"{name}: {inp}")

def main():
    port = 8080
    print("=== AutoChat Server v1.0 ===")
    print(f"Starting on port {port}...")
    server_in = queue.Queue()
    server_out = queue.Queue()
    t_server = threading.Thread(target=server, args=(server_in, server_out))
    t_server.daemon = True
    t_server.start()

    t1.daemon = True
    t2.daemon = True
    t1.start()
    t2.start()

    t1.join(timeout=2.0)
    t2.join(timeout=2.0)
```

The complete server has several improvements over the earlier versions:

1. **Name collision detection** -- the `Connect` handler checks
   `clients.contains(name)` and rejects duplicate usernames. This is a real
   validation concern that every chat server must handle.

2. **Broadcast and broadcast_room** -- helper functions that deliver messages

   server, these would iterate over the client list and send a message to each
   one.

   personal "Welcome" message via `send caller`, and broadcasts their arrival
   to everyone else.

4. **Error handling for private messages** -- the `PrivateMsg` handler checks

5. **Client commands** -- the `client_handler` now supports `/join`, `/leave`,

## Taking It Further

This chat server is a solid foundation. Here are some ideas for extending it:

1. **Authentication** -- add a `Login(String, String)` message that checks
   credentials before allowing a connection. Store passwords as hashes using
   `std::crypto`.

2. **Persistence** -- save chat history to a file using `write_file` from
   `std::fs`. Load it back on server restart with `read_file`.

3. **WebSocket support** -- use `std::net` to accept WebSocket connections
   instead of raw TCP. This lets browsers connect directly.

4. **Admin commands** -- add `/kick`, `/ban`, and `/create-room` commands
   that only administrators can use.

5. **Message history** -- store the last N messages per room and send them to
   new users when they join.

6. **Typing indicators** -- add a `Typing(String, String)` message that shows
   "Alice is typing..." to other room members.

7. **Rate limiting** -- prevent spam by limiting how many messages a user
   can send per second.

8. **Encryption** -- encrypt private messages end-to-end using `std::crypto`
   so that even the server cannot read them.

Each of these extensions exercises different parts of Auto's standard library
and type system. They are excellent practice projects.

Key takeaways:

1. **Design the protocol first** -- the `Message` enum is the contract
   between all components. Get it right, and everything else follows.

2. **One actor owns the state** -- the server actor is the single source of
   truth for clients and rooms. This eliminates an entire class of
   concurrency bugs.

3. **Build incrementally** -- we started with message types, then the server,
   then client handlers, then rooms, then the complete program. Each step was
   testable on its own.

4. **The actor model scales naturally** -- adding more features (auth,
   persistence, admin commands) means adding more message types and more
   handlers. The architecture stays the same.

5. **Error handling is not an afterthought** -- we check for duplicate names,
   missing users, and nonexistent rooms. These checks are part of the normal
   control flow, not bolted on afterward.
