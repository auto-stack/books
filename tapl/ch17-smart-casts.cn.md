# 智能类型转换与流式类型推导

在[第 7 章][ch07]中，我们学习了枚举和 `is` 模式匹配。在[第 13 章][ch13]中，我们
探索了泛型和规格（spec）。这些特性赋予了 Auto 强大的类型系统，但还缺少一块拼
图：当你知道一个值是几种可能性中的某一种时，如何安全地使用它？

这就是_智能类型转换（smart casts）_和_流式类型推导（flow typing）_——编译器根据
控制流自动缩小值类型的能力。如果你检查了 `if value is int`，编译器就知道在那个分
支内，`value` 就是一个 `int`。你不需要显式转换。你不需要类型标注。编译器会自动
为你处理。

在本章中，我们将学习 Auto 的流敏感类型系统如何工作，联合类型如何表达"几种类型之
一"，以及智能类型转换如何在保持代码安全的同时消除样板代码。

## 什么是智能类型转换？

_智能类型转换_是编译器根据控制流自动缩小值类型的能力。类型检查之后，编译器会记
住结果，并将该值视为缩小后的类型——无需显式转换。

许多语言都有这个概念。Kotlin 称之为"智能类型转换"。TypeScript 有"类型缩小"。
C# 有"基于模式的匹配"。Scala 3 有"匹配类型"。Auto 采用相同的思想，并将其与
`is` 关键字、联合类型和 `is` 表达式集成在一起。

核心思想很简单：如果你检查了 `if value is T`，那么在那个分支内，`value` _就是_
`T`。编译器信任你的检查，并相应地缩小类型。

为什么这很重要？三个原因：

1. **安全性** —— 你永远不会意外地将值用作错误的类型。编译器确保每个分支只看到它
   检查过的类型。

2. **便利性** —— 你不需要写显式转换、`as` 操作符或 `unwrap()` 调用。编译器为你
   完成这些工作。

3. **穷尽性** —— 编译器可以验证你处理了每种可能的类型。如果你遗漏了一种，它会
   在编译时（而非运行时）告诉你。

## `if x is T` 缩小

最简单的智能类型转换形式是 `if value is T`。这会检查 `value` 是否为类型 `T`，
并在该分支内缩小类型。

<Listing number="17-1" file-name="main.at" caption="使用 `if value is T` 进行智能类型转换">

```auto
fn describe(value int | String) {
    if value is int {
        print("Integer: " + value)
    } else {
        print("String: " + value)
    }
}

fn main() {
    describe(42)
    describe("hello")
}
```

让我们分解发生了什么：

1. **`fn describe(value int | String)`** —— 参数 `value` 的类型是 `int | String`，
   这是一个_联合类型_。它表示"value 是一个 `int` 或一个 `String`"。我们将在下一
   节详细探讨联合类型。

2. **`if value is int`** —— 检查 `value` 是否为 `int`。如果是，编译器在 `if` 分支
   内将 `value` 的类型缩小为 `int`。

3. **`print("Integer: " + value)`** —— 在 `if` 分支内，`value` 被确认为 `int`。
   `+` 操作符有效，因为 `int` 支持与 `String` 的拼接。

4. **`else`** —— 在 `else` 分支中，编译器将 `value` 的类型缩小为 `String`（联合
   类型中唯一剩余的类型）。

5. **`print("String: " + value)`** —— 在 `else` 分支内，`value` 被确认为
   `String`。字符串拼接直接有效。

智能类型转换消除了显式转换的需要。在许多语言中，你需要写类似 `(value as int)` 或
`value.asInstanceOf[Int]` 或 `dynamic_cast<int>(value)` 的代码。在 Auto 中，`is`
检查就足够了。

在 Python 中，等价的是 `isinstance(value, int)`。在 TypeScript 中，等价的是
`typeof value === "number"`。在 C 中，你需要一个带手动判别值检查的标签联合。在
Rust 中，你会对枚举使用 `match`。Auto 的 `is` 检查是其中最简洁的。

## 联合类型

联合类型是智能类型转换的基础。联合类型 `T | U` 表示"一个 `T` 类型的值或 `U`
类型的值"。你可以链式组合联合：`T | U | V` 表示"T、U 或 V 之一"。

Auto 中的联合类型等同于类型论中的和类型（sum types）。它们让你表达一个值可以是
几种不同事物之一，编译器会确保你处理每种可能性。

    process(3.14)

几个注意点：

1. **`int | float | String`** —— 三路联合类型。`|` 操作符组合类型。你可以有任意
   多个选择。

2. **`print(value)`** —— `print` 函数接受任何支持格式化显示的值。由于 `int`、
   `float` 和 `String` 都支持打印，这里无需缩小类型。编译器通过共享接口解析调用。

3. **`process(42)`** —— Auto 推断 `42` 是 `int`，满足 `int | float | String` 联合
   类型。无需标注。

4. **`process(3.14)`** —— 类似地，`3.14` 被推断为 `float`，是联合类型的一部分。

5. **`process("hello")`** —— 字符串字面量满足联合类型中的 `String` 部分。

联合类型是 Auto 对"这个参数应该是什么类型？"这一问题的回答。你不需要使用基类或
接口，只需列出允许的类型。编译器确保只有这些类型的值被传入。

在 TypeScript 中，联合类型的工作方式相同：`number | string`。在 Python 中，你会
使用 `typing` 模块的 `Union[int, float, str]`。在 Rust 中，你会定义一个枚举，为
每种类型创建一个变体。在 C 中，你会使用标签联合。Auto 的 `T | U` 语法是这一思想
最直接的表达。

## 穷尽缩小

当联合类型有一组有限的选项时，编译器可以检查你是否处理了每种情况。这被称为_穷尽
匹配_，它同时适用于 `if/else` 链和 `is` 表达式。

枚举（来自[第 7 章][ch07]）是穷尽缩小的天然选择。每个变体都是不同的类型，编译器
知道完整的变体集合。

<Listing number="17-3" file-name="main.at" caption="使用枚举模式匹配进行穷尽缩小">

```auto
enum Shape {
    Circle(float),
    Rect(float, float),
    Triangle(float, float, float)
}

fn area(shape Shape) float {
    is shape {
        Circle(r) => 3.14159 * r * r,
        Rect(w, h) => w * h,
        Triangle(a, b, c) => 0.0 // simplified
    }
}

fn main() {
    var c = Shape.Circle(5.0)
    print(area(c))
}
```

    return 0.0;

要点：

1. **`enum Shape`** —— 定义三个变体，每个携带不同的数据。`Circle` 有一个 `float`
   （半径）。`Rect` 有两个 `float`（宽度和高度）。`Triangle` 有三个 `float`（边长）。

2. **`is shape { Circle(r) => ..., Rect(w, h) => ..., Triangle(a, b, c) => ... }`**
   —— `is` 表达式执行穷尽模式匹配。每个变体被解构，内部值绑定到局部变量。

3. **`Circle(r) => 3.14159 * r * r`** —— 在这个分支内，`r` 是一个 `float`（从
   `Circle(float)` 变体缩小而来）。无需转换。

4. **`Rect(w, h) => w * h`** —— 类似地，`w` 和 `h` 在这个分支内被缩小为 `float`。

5. **`Triangle(a, b, c) => 0.0`** —— 三角形的情况这里做了简化。在实际实现中，你会
   使用海伦公式：`sqrt(s * (s-a) * (s-b) * (s-c))`，其中 `s = (a + b + c) / 2`。

6. **`Shape.Circle(5.0)`** —— 枚举变体构造使用点号，不是 `::`（来自[第 7 章][ch07]）。

穷尽性保证非常强大。如果你向 `Shape` 添加一个新变体——比如 `Square(float)` —— 编译器
会标记 `area` 为非穷尽的。你不可能遗漏任何一种情况。

在 Rust 中，这是相同的 `match` 表达式。在 Python 3.10+ 中，这是 `match/case` 语句。
在 C 中，这是带标签联合的 `switch`。在 TypeScript 中，你会使用带判别值的联合和
`switch`。Auto 的 `is` 表达式是这种模式最自然的语法。

## 条件中的智能类型转换

智能类型转换不限于简单的类型检查。它们适用于方法调用、嵌套条件以及任何流敏感的
上下文。成功的 `is` 检查之后，缩小后的类型会在分支的剩余部分持续有效。

```auto

    if v is String {
        v.to_upper()
    } else {
        "Not a string"
    }
}

fn main() {
    print(format_value("hello"))
    print(format_value(42))
}

#include <stdio.h>
#include <stdbool.h>
#include <string.h>
#include <ctype.h>

typedef enum ValueType {
    ValueType_Int,
    ValueType_Float,
    ValueType_String,
} ValueType;

typedef struct {
    ValueType kind;
    union {
        long int_val;
        double float_val;
        const char* str_val;
    };
} Value;

const char* format_value(Value v) {
    if (v.kind == ValueType_String) {
        static char buf[256];
        strcpy(buf, v.str_val);
        for (int i = 0; buf[i]; i++) {
            buf[i] = toupper((unsigned char)buf[i]);
        }
        return buf;
    } else {
        return "Not a string";
    }
}

int main(void) {
    printf("%s\n", format_value((Value){ .kind = ValueType_String, .str_val = "hello" }));
    printf("%s\n", format_value((Value){ .kind = ValueType_Int, .int_val = 42 }));
    return 0;
}
```

关键观察：

1. **`v.to_upper()`** —— 这个方法调用只对 `String` 值有意义。但 `v` 被声明为
   `int | String | float`。我们怎么能对它调用 `.to_upper()` 呢？

2. **智能类型转换** —— 在 `if v is String` 之后，编译器在 `if` 分支内将 `v` 缩小
   为 `String`。现在 `.to_upper()` 是有效的调用。编译器知道 `v` 是 `String`，因此
   允许调用 `String` 方法。

3. **`else`** —— 在 `else` 分支中，`v` 被缩小为 `int | float`（所有不是 `String`
   的类型）。`int` 和 `float` 都没有 `.to_upper()` 方法，因此代码返回一个回退
   字符串。

这就是智能类型转换变得关键的地方。没有它，你需要在调用 `.to_upper()` 之前将 `v`
显式转换为 `String`。有了智能类型转换，编译器根据 `is` 检查自动完成缩小。

在 TypeScript 中，这被称为"类型缩小"，通过 `typeof` 检查和类型守卫实现。在 Kotlin
中，这是"智能类型转换"特性。在 Python 中，`isinstance` 不提供静态缩小（mypy 可以
推断，但运行时不强制执行）。Auto 的智能类型转换在编译时进行检查。

## 类型安全分派

智能类型转换和联合类型处理"几种类型之一"的情况。但多态呢——当你想通过共享接口
统一处理不同类型时？这就是_规格（spec）_发挥作用的地方（来自[第 13 章][ch13]）。

`spec` 定义了一组类型必须实现的方法。当一个类型符合某个 spec 时，你可以在任何需要
该 spec 的地方使用该类型的值。这是 Auto 版本的接口（interface）、特征（trait）或
协议（protocol）。

<Listing number="17-5" file-name="main.at" caption="通过规格实现类型安全分派">

```auto
spec Renderable {
    fn render(&self) String
}

type Button {
    label String
}

ext Button as Renderable {
    fn render(&self) String {
        "[Button: " + self.label + "]"
    }
}

type TextField {
    text String
}

ext TextField as Renderable {
    fn render(&self) String {
        "[TextField: " + self.text + "]"
    }
}

fn render_all(items []Renderable) {
    for item in items {
        print(item.render())
    }
}

fn main() {
    var b = Button { label: "Click me" }
    var t = TextField { text: "Type here" }
    render_all([b, t])
}
```

1. **`spec Renderable`** —— 定义一个契约：任何作为 `Renderable` 的类型必须有一个
   返回 `String` 的 `render` 方法。这与 Rust 的 `trait`、Java 的 `interface` 或

2. **`ext Button as Renderable`** —— 声明 `Button` 符合 `Renderable` 规格。`as`
   关键字使符合性明确。`ext` 块提供实现。这是 Auto 中 `impl Renderable for Button`
   的等价写法。

3. **`fn render(&self) String`** —— 方法接受 `&self`（实例的引用）并返回一个
   `String`。在方法内部，`self.label` 访问 `Button` 的 `label` 字段。

4. **`fn render_all(items []Renderable)`** —— 这个函数接受一个 `Renderable` 值的列
   表。它不在乎它们是 `Button` 还是 `TextField`——它只在乎它们实现了 `render()`。

5. **`render_all([b, t])`** —— Auto 在需要时自动将 `Button` 和 `TextField` 转换为
   `Renderable`。两种类型都满足规格，因此列表是有效的。

这种方法与智能类型转换的关键区别：

- **智能类型转换** —— 你在运行时检查类型并分支。编译器在每个分支中缩小类型。最
  适合每种类型需要不同行为的场景。

- **规格（分派）** —— 你定义一个共享接口并通过它调用方法。运行时分派到正确的实现。
  最适合不同类型共享相同概念但实现不同的场景。

当你需要针对每种类型不同行为时，使用智能类型转换。当不同类型共享一个行为时，使用
规格来定义接口，使用 `ext ... as` 来实现它。在实践中，设计良好的 Auto 代码两者兼
用：规格用于共享接口，智能类型转换用于需要类型特定细节的场景。

在 Rust 中，这个模式是 `trait` + `impl Trait for Type`。在 TypeScript 中，是
`interface` + `implements`。在 Python 中，是 `ABC` + 继承。在 C 中，是虚函数表中
的函数指针（这正是转译器生成的内容）。Auto 的 `spec` + `ext ... as` 是表达这种
模式最简洁的方式。

## 总结

本章介绍了 Auto 中的智能类型转换和流式类型推导。以下是我们学到的内容：

| 概念 | 语法 | 用途 |
|------|------|------|
| 联合类型 | `T \| U` | 一个值是几种类型之一 |
| 类型检查 | `if x is T` | 检查并缩小值的类型 |
| 穷尽匹配 | `is x { A => ..., B => ... }` | 处理枚举的每个变体 |
| 规格 | `spec Name { ... }` | 定义共享接口 |
| 规格符合 | `ext Type as Spec { ... }` | 为类型实现规格 |
| 规格分派 | `fn foo(x Spec)` | 通过共享接口调用方法 |

核心要点：

1. **智能类型转换是自动的** —— 在 `if value is T` 之后，编译器在该分支内将
   `value` 缩小为 `T`。无需显式转换。

2. **联合类型表达可选性** —— `int | String` 表示"要么是 int 要么是 String"。编译器
   跟踪代码中每个点可能的类型。

3. **穷尽性被强制执行** —— 对枚举进行模式匹配时，编译器确保每个变体都被处理。添加
   新变体是安全的，因为编译器会标记不完整的匹配。

4. **智能类型转换启用类型特定方法** —— 缩小之后，你可以调用只在缩小后的类型上存在
   的方法。

5. **规格提供多态** —— 当不同类型共享一个行为时，使用 `spec` 定义接口，使用
   `ext ... as` 实现。这比按类型分支更简洁。

智能类型转换和规格共同赋予了 Auto 一个既安全又富有表现力的类型系统。智能类型转换
处理"我知道这个值是什么类型"的情况。规格处理"我不在乎这个值是什么类型，只要它支持
这个行为"的情况。每个设计良好的程序都需要这两者。
