# Project: File Processor

This chapter is the capstone of Phase 2. We will build a complete CLI file
processor -- a small tool that reads a text file, searches for lines matching a
pattern, and reports the results. Along the way, we will put together every
concept from Phase 2: custom types, enums, error handling with `!T`, file I/O,
and generics.

Our file processor will combine:

- **Custom types** ([Chapter 10][ch10]) -- a `Config` type for program settings
- **Enums and pattern matching** ([Chapter 11][ch11]) -- modeling outcomes
- **Error handling with `!T`** ([Chapter 12][ch12]) -- propagating and handling
  errors with the bang type
- **File I/O** ([Chapter 12][ch12]) -- reading file contents as strings
- **Generics** ([Chapter 13][ch13]) -- writing reusable search with type
  parameters and higher-order functions

## Setting Up

Here is what we are building: a command-line tool called `fileproc` that accepts
an input file path, an output file path, and a verbose flag. It reads the input
file, searches each line for the word "hello", counts the matches, and
optionally prints each matching line.

By the end of this chapter, the program will:

1. Parse command-line arguments into a `Config` type
2. Validate the configuration, rejecting invalid inputs
3. Read file contents with proper error handling
4. Search lines using a generic, higher-order function
5. Print results with optional verbose output

We will build this step by step, adding one concept at a time.

## Parsing Arguments

The first step in any CLI program is gathering configuration from the user.
Rather than scattering individual variables throughout the code, we group all
settings into a single `Config` type. This makes the code easier to read,
easier to pass around, and easier to extend later.

The `Config` type has three fields: two strings for file paths and a boolean for
verbose mode. The `parse_args` function constructs a `Config` with default values.
In a real program, this function would read `env.args()` to get command-line
arguments -- for now we use hardcoded defaults to keep things simple.

This pattern -- grouping related values into a type -- is one of the most
important organizational tools in Auto. As the program grows, you can add new
fields to `Config` without changing how it is passed around.

The `read_file` function returns `!String`, which is Auto's Result type. The `!`
prefix means "this operation might fail." On success, it returns the file
contents as a `String`. On failure, it returns an error message.

The caller unwraps the result with `!` (the postfix bang operator, equivalent to
Rust's `?`). If `read_file` fails, the error is propagated up immediately. If it
succeeds, `content` holds the file contents as a plain `String`.

This is the same pattern we saw in [Chapter 12][ch12]. The key difference from
languages like Python or JavaScript is that Auto _forces_ you to acknowledge the
possibility of failure. You cannot accidentally ignore an error -- the compiler
requires the `!` unwrap.

In C, the transpiler generates a tagged union (`ResultString`) with an `is_ok`
flag. The caller must check the flag before accessing the value. This is exactly
what Auto does internally, but the compiler handles it automatically.

## Validating Configuration

Before processing, we should validate the config. An empty input path makes no
sense -- we should reject it early with a clear error message. This is where
`!T` really shines: validation errors propagate through the same mechanism as
I/O errors.

The `validate` function returns `!Config` -- either the same config (if valid)
or an error string (if invalid). In Auto, the last expression in a function body
is the return value. The `if/else` expression returns `"input path cannot be
empty"` on the error branch and `c` (the original config) on the success branch.

This is a powerful pattern. The return type `!Config` means the function can
either succeed with a `Config` or fail with a `String`. The caller uses `!` to
unwrap, propagating any error. Because both `validate` and `read_file` return
`!T`, they chain naturally: each `!` in the call stack propagates errors
upward until `main` handles them.

How this maps across languages:

- **Rust** uses `Result<Config, String>` with `Ok(c)` and `Err(msg)`. The `?`
  operator unwraps and propagates errors.
- **Python** raises a `ValidationError` exception. There is no compile-time
  requirement to handle it.
- **C** uses a tagged union `ResultConfig` with manual `is_ok` checking.
- **TypeScript** throws an `Error`. Like Python, error handling is optional.

## Generic Search

Now we arrive at the core of our processor: searching lines. We want a
reusable function that takes a list of strings and a predicate (a function that
decides whether a line matches), and returns the matching results.

This is a perfect use case for generics combined with higher-order functions.
The search logic is the same regardless of what the predicate returns -- it
could return a boolean, a filtered string, a count, or anything else.

<Listing number="14-4" file-name="main.at" caption="A generic `search_lines` function with a higher-order predicate">

```auto
fn search_lines<T>(lines []String, predicate fn(String) T) []T {
    var results = []
    for line in lines {
        var result = predicate(line)
        results.push(result)
    }
    results
}

fn main() {
    var lines = ["hello world", "foo bar", "hello auto"]
    var found = search_lines(lines, fn(s String) ?String {
        if s.contains("hello") { s } else { none }
    })
    for f in found {
        print(f)
    }
}
```

Let us break down the key parts:

1. **`fn search_lines<T>`** -- The function is generic over type `T`. The
   predicate decides what `T` is.

2. **`predicate fn(String) T`** -- The second parameter is itself a function.
   It takes a `String` and returns a `T`. This is a _higher-order function_ --
   a function that takes another function as an argument.

3. **Anonymous function** -- The predicate is defined inline:
   `fn(s String) ?String { ... }`. This is Auto's syntax for anonymous
   functions (similar to Rust closures or JavaScript arrow functions). The
   return type `?String` means it returns either a matching string or `none`.

4. **`s.contains("hello")`** -- String method that checks if the substring
   exists.

The predicate returns `?String` (Option). When the line contains "hello", it
returns `s` (the line itself). Otherwise it returns `none`. The `search_lines`
function collects all non-`none` results into the output list.

This design is deeply flexible. You could pass a different predicate to count
matches, extract email addresses, find lines longer than 80 characters, or
anything else -- all without changing `search_lines`. The generic type parameter
`T` ensures that the function works with any return type the predicate
produces.

In Rust, this maps to closures (`|s: &String| -> Option<String>`). In Python,
it is a `lambda`. In TypeScript, an arrow function. In C, the transpiler
generates a concrete function per predicate because C has no closures.

## The Complete Processor

Now we bring everything together. The complete file processor combines the
`Config` type, file reading with error handling, line iteration, and verbose
output into a single, clean program.

<Listing number="14-5" file-name="main.at" caption="The complete file processor -- combining types, error handling, and file I/O">

```auto

    input_path String
    output_path String
```

```rust
struct Config {
    input_path: String,
    output_path: String,
    verbose: bool,
}

    Ok(String::from("line1\nline2\nhello world\nline3\nhello auto"))

fn process(config: &Config) -> Result<i64, String> {
    let content = read_file(&config.input_path)?;
    let lines: Vec<String> = content.split('\n').map(|s| String::from(s)).collect();

                println!("{}", "Found: ".to_string() + line);
            }
        }
    }
    Ok(count)
}

    let config = Config { input_path: String::from("data.txt"), output_path: String::from("result.txt"), verbose: true };
```

```c
#include <stdio.h>
#include <stdbool.h>
#include <string.h>

typedef struct Config {
    const char* input_path;
    const char* output_path;
    bool verbose;
} Config;

const char* get_file_content(void) {
    return "line1\nline2\nhello world\nline3\nhello auto";
}

long long process(Config config) {
    const char* content = get_file_content();
    char lines[16][256];
    int line_count = 0;
    char buf[1024];
    strncpy(buf, content, sizeof(buf) - 1);
    buf[sizeof(buf) - 1] = '\0';

    char* token = strtok(buf, "\n");
    while (token != NULL && line_count < 16) {
        strncpy(lines[line_count], token, 255);
        lines[line_count][255] = '\0';
        line_count++;
        token = strtok(NULL, "\n");
    }

    long long count = 0;
    for (int i = 0; i < line_count; i++) {
        if (strstr(lines[i], "hello") != NULL) {
            count = count + 1;
            if (config.verbose) {
                printf("Found: %s\n", lines[i]);
            }
        }
    }

    return count;
}

int main(void) {
    Config config = { .input_path = "data.txt", .output_path = "result.txt", .verbose = true };
    long long matches = process(config);
    printf("Found %lld matches\n", matches);
    return 0;
}
```

Let us trace through the execution:

1. **`main`** creates a `Config` with `verbose: true` and calls `process`.
2. **`process`** calls `read_file`, which returns simulated file content. The `!`
   unwrap propagates any error.
3. **`content.split("\n")`** breaks the string into an array of lines.
4. The `for` loop checks each line. When `line.contains("hello")` is true, it

5. The function returns `count`, which `main` prints as the final summary.

Notice how clean the error handling is. Both `read_file` and `process` return
`!T`. The `!` operator at each call site propagates errors upward. If `main`
wanted to handle errors itself, it could use an `is` expression instead of `!`:

```auto
is read_file("data.txt") {
    Ok(content) => print(content),
    Err(msg) => eprint("Error: " + msg)
}
```

This is the same `Result` pattern from [Chapter 12][ch12], now applied in a
real project context.

## Summary

This capstone project brought together every concept from Phase 2:

| Concept | Where We Used It |
|---------|-----------------|
| Custom types | `Config` type groups all settings |
| Field access | `config.input_path`, `config.verbose` |
| Error handling (`!T`) | `read_file` and `process` return `!T` |
| Error propagation | `!` unwrap at each call site |
| String methods | `s.contains("hello")`, `content.split("\n")` |
| Control flow | `if/else` for validation and verbose output |
| For loops | Iterating over lines |
| Generics | `search_lines<T>` with type parameter |
| Higher-order functions | `predicate fn(String) T` parameter |
| Anonymous functions | Inline `fn(s String) ?String { ... }` |
| Option type (`?T`) | Predicate returns `?String` for filtering |

Here is what we learned across Phase 2:

1. **Types** (Chapter 10) give us structured data with named fields. `Config`
   is the backbone of our program.

2. **Enums** (Chapter 11) model data that can be one of several variants. The
   `?T` (Option) and `!T` (Result) types are enums under the hood.

3. **Error handling** (Chapter 12) uses `!T` to make failure explicit. The `!`
   unwrap operator propagates errors cleanly. There are no surprise crashes.

4. **Generics** (Chapter 13) let us write functions that work across types.
   `search_lines<T>` is reusable with any predicate return type.
