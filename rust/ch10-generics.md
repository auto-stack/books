# Generic Types, Specs, and Lifetimes

Every programming language has tools for effectively handling the duplication
of concepts. In Auto, one such tool is _generics_: abstract stand-ins for
concrete types or other properties. We can express the behavior of generics or
how they relate to other generics without knowing what will be in their place
when compiling and running the code.

Functions can take parameters of some generic type, instead of a concrete type
like `int` or `String`, in the same way they take parameters with unknown
values to run the same code on multiple concrete values. In fact, we already
used generics in earlier chapters: `?T` (Option), `!T` (Result), `List<T>`,
and `Map<K, V>` are all generic types.

In this chapter, we'll explore how to define your own types, functions, and
methods with generics. Then we'll learn about _specs_ (Auto's equivalent of
Rust traits) for defining shared behavior. Finally, we'll briefly discuss
lifetimes — and how Auto's compiler handles them automatically, eliminating
the need for explicit lifetime annotations.

## Generic Data Types

### In Function Definitions

When defining a function that uses generics, we place the generics in the
signature where we would usually specify the data types. This makes our code
more flexible while preventing duplication.

<Listing number="10-1" file-name="main.auto" caption="A generic function to find the largest value in a list">

```auto
fn largest<T>(list []T) T {
    var largest = list[0]
    for item in list {
        if item > largest {
            largest = item
        }
    }
    largest
}

fn main() {
    let numbers = [34, 50, 25, 100, 65]
    let result = largest(numbers)
    print(f"The largest number is ${result}")
}
```


</Listing>

We read `fn largest<T>(list []T) T` as "the function `largest` is generic over
some type `T`." The function takes a list of `T` values and returns a `T`. By
convention, type parameter names are short, often just one letter, using
UpperCamelCase — `T` (short for "type") is the default choice.

The syntax is the same as Rust's: type parameters go inside angle brackets `<>`
between the function name and the parameter list.

### In Type Definitions

We can define types with generic type parameters in their fields:

<Listing number="10-2" file-name="main.auto" caption="A `Point<T>` type that holds `x` and `y` values of type `T`">

```auto
type Point<T> {
    x T
    y T
}

fn main() {
    let integer = Point(5, 10)
    let float = Point(1.0, 4.0)
    print(f"integer: (${integer.x}, ${integer.y})")
    print(f"float: (${float.x}, ${float.y})")
}
```

```rust
struct Point<T> {
    x: T,
    y: T,
}

fn main() {
    let integer = Point { x: 5, y: 10 };
    let float = Point { x: 1.0, y: 4.0 };
    println!("integer: ({}, {})", integer.x, integer.y);
    println!("float: ({}, {})", float.x, float.y);
}
```

</Listing>

Because we've used only one generic type `T`, both `x` and `y` must be the
same type. If you need different types for each field, use multiple type
parameters:

<Listing number="10-3" file-name="main.auto" caption="A `Point<T, U>` with two type parameters">

```auto
type Point<T, U> {
    x T
    y U
}

fn main() {
    let p1 = Point(5, 10.4)
    let p2 = Point("Hello", 'c')
    print(f"p1: (${p1.x}, ${p1.y})")
    print(f"p2: (${p2.x}, ${p2.y})")
}
```

```rust
struct Point<T, U> {
    x: T,
    y: U,
}

fn main() {
    let p1 = Point { x: 5, y: 10.4 };
    let p2 = Point { x: String::from("Hello"), y: 'c' };
    println!("p1: ({}, {})", p1.x, p1.y);
    println!("p2: ({}, {})", p2.x, p2.y);
}
```

</Listing>

### In Method Definitions (ext)

We can define methods on generic types using `ext`:

<Listing number="10-4" file-name="main.auto" caption="Implementing a method on `Point<T>` with `ext`">

```auto
type Point<T> {
    x T
    y T
}

ext Point<T> {
    fn x_value(self) T {
        self.x
    }
}

fn main() {
    let p = Point(5, 10)
    let x = p.x_value()
    print(f"x = ${x}")
}
```

```rust
struct Point<T> {
    x: T,
    y: T,
}

impl<T> Point<T> {
    fn x_value(&self) -> T {
        self.x.clone()
    }
}

fn main() {
    let p = Point { x: 5, y: 10 };
    let x = p.x_value();
    println!("x = {}", x);
}
```

</Listing>

Note that we declare `T` after `ext` and before `Point<T>`, just like Rust's
`impl<T> Point<T>`. This tells the compiler that the method is defined for any
concrete type that `T` might be.

### Performance of Code Using Generics

Using generics has zero runtime cost. Auto (like Rust) uses _monomorphization_:
the compiler generates specific code for each concrete type used with a generic.
So `Point<int>` and `Point<f64>` become separate, optimized types at compile
time. There is no runtime overhead for using generics.

## Defining Shared Behavior with Specs

A _spec_ (short for "specification") defines the functionality a particular
type has and can share with other types. Specs are Auto's equivalent of Rust's
_traits_ (also known as _interfaces_ in other languages).

### Defining a Spec

A type's behavior consists of the methods we can call on that type. Spec
definitions group method signatures together to define a set of behaviors:

```auto
spec Summary {
    fn summarize(self) String
}
```

This declares a spec named `Summary` with one required method: `summarize`,
which takes `self` and returns a `String`. Any type that implements this spec
must provide its own `summarize` method.

### Implementing a Spec on a Type

In Auto, you implement a spec using `as`:

<Listing number="10-5" file-name="main.auto" caption="Defining a spec and implementing it on two types">

```auto
spec Summary {
    fn summarize(self) String
}

type NewsArticle as Summary {
    headline String
    location String
    author String
    content String
}

ext NewsArticle {
    fn summarize(self) String {
        f"${self.headline}, by ${self.author} (${self.location})"
    }
}

type Tweet as Summary {
    username String
    content String
}

ext Tweet {
    fn summarize(self) String {
        f"${self.username}: ${self.content}"
    }
}

fn main() {
    let article = NewsArticle("Headline", "NYC", "Author", "Content")
    let tweet = Tweet("user", "hello world")
    print(article.summarize())
    print(tweet.summarize())
}
```

```rust

    fn summarize(&self) -> String;

    author: String,

}

    fn summarize(&self) -> String {
        format!("{}, by {} ({})", self.headline, self.author, self.location)
    }
}

    username: String,
    content: String,
}

    fn summarize(&self) -> String {
        format!("{}: {}", self.username, self.content)
    }
}

fn main() {
    let article = NewsArticle { headline: String::from("Headline"), location: String::from("NYC"), author: String::from("Author"), content: String::from("Content") };
    let tweet = Tweet { username: String::from("user"), content: String::from("hello world") };
    println!("{}", article.summarize());
    println!("{}", tweet.summarize());
}
```

</Listing>

Key differences from Rust:

| Feature | Auto | Rust |
|---------|------|------|
| Define spec/trait | `spec Summary { ... }` | `trait Summary { ... }` |
| Implement on type | `type Foo as Summary { ... }` | `impl Summary for Foo { ... }` |
| Method body | In separate `ext` block | Inside `impl` block |

The `as` keyword on the type declaration means "this type conforms to the
spec." The actual method implementations go in the `ext` block.

### Using Specs as Parameters

You can use a spec as a parameter type to accept any type that implements it:

<Listing number="10-6" file-name="main.auto" caption="Using a spec as a function parameter">

```auto
spec Summary {
    fn summarize(self) String
}

type NewsArticle as Summary {
    headline String
    author String
}

ext NewsArticle {
    fn summarize(self) String {
        f"${self.headline}, by ${self.author}"
    }
}

fn notify(item Summary) {
    print("Breaking news!")
    print(item.summarize())
}

fn main() {
    let article = NewsArticle("Big News", "Alice")
    notify(article)
}
```

```rust
// Auto-generated by a2r transpiler

// a2r Standard Library (from crate)
#[allow(unused_imports)]
use auto_lang::a2r_std::*;

trait Summary {
    fn summarize(&self) -> String;
}

struct NewsArticle {
    headline: String,
    author: String,
}

impl Summary for NewsArticle {
    fn summarize(&self) -> String {
        format!("{}, by {}", self.headline, self.author)
    }
}

fn notify(item: Box<dyn Summary>) {
    println!("Breaking news!");
    println!("{}", item.summarize());
}

fn main() {
    let article = NewsArticle { headline: String::from("Big News"), author: String::from("Alice") };
    notify(Box::new(article));
}
```

</Listing>

In Auto, `fn notify(item Summary)` accepts any value whose type implements the
`Summary` spec. This is equivalent to Rust's `impl Trait` syntax:
`fn notify(item: impl Summary)`.

Note how Auto's syntax is simpler — you don't need `impl` or `Box<dyn Trait>`
to pass spec-conforming values. The compiler handles the dispatch automatically.

### Generic Specs

Specs can have their own type parameters, just like types:

<Listing number="10-7" file-name="main.auto" caption="A generic spec with a type parameter">

```auto
spec Container<T> {
    fn get(self) T
    fn set(self, value T)
}

type Box<T> as Container<T> {
    value T
}

ext Box<T> {
    fn get(self) T {
        self.value
    }
    fn set(self, value T) {
        self.value = value
    }
}

fn main() {
    var b = Box(42)
    let v = b.get()
    print(f"Value: ${v}")
    b.set(100)
    print(f"Updated: ${b.get()}")
}
```

```rust
// Auto-generated by a2r transpiler

// a2r Standard Library (from crate)
#[allow(unused_imports)]
use auto_lang::a2r_std::*;

trait Container<T> {
    fn get(&self) -> T;
    fn set(&mut self, value: T);
}

struct Box<T> {
    value: T,
}

impl<T> Container<T> for Box<T> {
    fn get(&self) -> T {
        self.value.clone()
    }
    fn set(&mut self, value: T) {
        self.value = value;
    }
}

fn main() {
    let mut b = Box { value: 42 };
    let v = b.get();
    println!("Value: {}", v);
    b.set(100);
    println!("Updated: {}", b.get());
}
```

</Listing>

The generic spec `Container<T>` defines a contract that any implementing type
must provide `get` and `set` methods for its inner type `T`. The `Box<T>` type
implements `Container<T>`, and the `ext` block provides the method bodies.

> #### Spec vs Trait Quick Reference
>
> | Auto | Rust | Meaning |
> |------|------|---------|
> | `spec Foo { ... }` | `trait Foo { ... }` | Define a specification |
> | `type Bar as Foo { ... }` | `impl Foo for Bar { ... }` | Implement spec on type |
> | `fn foo(x SomeSpec)` | `fn foo(x: impl SomeSpec)` | Spec as parameter |
> | `spec Container<T>` | `trait Container<T>` | Generic spec |
> | `ext Type<T> { ... }` | `impl<T> Type<T> { ... }` | Generic method extension |

## Lifetimes: Automatic in Auto

In Rust, _lifetimes_ are generic annotations that tell the compiler how long
references should be valid. They're necessary because Rust needs explicit
information about the relationships between references.

Auto takes a different approach: **lifetimes are fully automatic**. The compiler
tracks reference validity internally and ensures memory safety without requiring
you to write any lifetime annotations.

### No Lifetime Annotations Needed

In Rust, a function that returns the longer of two string slices requires
explicit lifetime annotations:

```rust
// Rust requires explicit lifetimes
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

In Auto, the same function needs no annotations at all:

<Listing number="10-8" file-name="main.auto" caption="Auto infers lifetimes automatically — no annotations needed">

```auto
fn longest(a String, b String) String {
    if a.len() > b.len() {
        a
    } else {
        b
    }
}

fn main() {
    let result = longest("long string", "xyz")
    print(f"The longest string is ${result}")
}
```

```rust
// Auto-generated by a2r transpiler

// a2r Standard Library (from crate)
#[allow(unused_imports)]
use auto_lang::a2r_std::*;

fn longest(a: String, b: String) -> String {
    if a.len() > b.len() {
        a
    } else {
        b
    }
}

fn main() {
    let result = longest(String::from("long string"), String::from("xyz"));
    println!("The longest string is {}", result);
}
```

</Listing>

Auto's compiler uses ownership and borrowing rules (`.view`, `.mut`, `.move`)
to track reference validity at compile time, eliminating the need for explicit
lifetime parameters in most cases.

### What This Means for You

- **No `'a` annotations** in function signatures
- **No lifetime parameters** on struct definitions
- **No `<'a>` declarations** in `ext` blocks
- The compiler still prevents dangling references and use-after-free bugs

If you're coming from Rust, you can think of it this way: Auto's compiler
applies Rust's lifetime elision rules aggressively, plus additional inference,
so you almost never need to write lifetimes explicitly.

## Summary

Auto provides the same generic programming capabilities as Rust, with a
cleaner syntax:

| Concept | Auto | Rust |
|---------|------|------|
| Generic function | `fn foo<T>(x T)` | `fn foo<T>(x: T)` |
| Generic type | `type Point<T> { x T }` | `struct Point<T> { x: T }` |
| Multiple type params | `type Pair<T, U>` | `struct Pair<T, U>` |
| Generic methods | `ext Point<T> { ... }` | `impl<T> Point<T> { ... }` |
| Spec (trait) | `spec Foo { ... }` | `trait Foo { ... }` |
| Implement spec | `type Bar as Foo { ... }` | `impl Foo for Bar { ... }` |
| Spec as parameter | `fn foo(x SomeSpec)` | `fn foo(x: impl SomeSpec)` |
| Generic spec | `spec Storage<T>` | `trait Storage<T>` |
| Lifetimes | Automatic | Explicit `'a` annotations |

The most notable simplification is Auto's automatic lifetime inference. Where
Rust requires `<'a>` annotations on function signatures and struct definitions,
Auto's compiler handles this internally, letting you focus on your logic rather
than lifetime management.
