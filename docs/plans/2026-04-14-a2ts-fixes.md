## Task 1: Fix `ext` self parameter handling

**Files:**
- Modify: `crates/auto-lang/src/trans/ts_stmt.rs:691-719` (ext_decl)
- Test: `crates/auto-lang/test/a2ts/033_ext/ext.at`
- Test: `crates/auto-lang/test/a2ts/033_ext/ext.expected.ts`

**Step 1: Understand the current behavior**

Current `ext_decl()` iterates ALL params including `self`:
```rust
for (i, param) in method.params.iter().enumerate() {
    if i > 0 { out.write(b", ")?; }
    out.write_all(param.name.as_bytes())?;  // outputs "self" as a parameter!
```

The Rust transpiler (`rust.rs:2229-2240`) handles this via `fn_decl()` which checks `is_method && !fn_decl.is_static` and auto-emits `&self` / `&mut self`, skipping `self` from the param list. But a2ts `ext_decl()` doesn't use `fn_decl()` — it manually iterates params.

**Step 2: Fix ext_decl to skip `self` parameter**

In `ts_stmt.rs`, modify `ext_decl()` to:
1. Skip the first param if its name is `"self"`
2. Do NOT output `self` in the parameter list (TypeScript methods use `this` implicitly)

```rust
pub fn ext_decl(&mut self, ext: &Ext, out: &mut impl Write) -> AutoResult<()> {
    for method in &ext.methods {
        out.write_all(ext.target.as_bytes())?;
        out.write(b".prototype.")?;
        out.write_all(method.name.as_bytes())?;
        out.write(b" = function(")?;

        let mut first = true;
        for param in method.params.iter() {
            if param.name == "self" {
                continue;  // Skip self — TypeScript uses implicit `this`
            }
            if !first { out.write(b", ")?; }
            first = false;
            out.write_all(param.name.as_bytes())?;
            if !matches!(param.ty, Type::Unknown) {
                out.write(b": ")?;
                out.write_all(Self::type_to_ts(&param.ty).as_bytes())?;
            }
        }
        // ... rest unchanged
```

## Task 2: Fix `ext` method body missing `return`

**Files:**
- Modify: `crates/auto-lang/src/trans/ts_stmt.rs:715` (ext_decl body)
- Test: Same as Task 1

**Step 1: Understand the current behavior**

Current `ext_decl()` calls `self.body(&method.body, out)` which outputs the body statements. The problem is that the last expression in a method body is not wrapped in `return`. In TypeScript, a method that should return a value needs an explicit `return` statement.

The Rust transpiler handles this via `fn_decl()` which has logic to detect the last expression and emit `return` for it. The a2ts `body()` method may or may not have this logic.

**Step 2: Check the body() method**

Look at `ts_stmt.rs` for the `body()` method implementation. If it already handles return for the last expression in non-void function contexts, then the fix is to pass the return type info to `body()`. If not, add return logic specifically for `ext_decl`.

**Step 3: Implement the fix**

Option A (preferred): If `body()` supports a "needs_return" parameter, pass `true` when the method return type is not `Void`.

Option B: After `self.body()`, check if the method's return type is non-void and if so, post-process the output to add `return` before the last expression.

Option C (simplest): Write a dedicated `ext_method_body()` helper that outputs the body with proper `return` handling — skip `self` references (they become `this`), and add `return` before the last expression when ret is non-void.

**Step 4: Verify with the test from Task 1**

Run: `cargo test -p auto-lang --lib test_033_ext`
Expected: PASS

## Task 3: Fix `Box`/`Arc` hardcoding in parser for a2ts context

**Files:**
- Modify: `crates/auto-lang/src/parser.rs:2516-2532`
- Modify: `crates/auto-lang/src/trans/ts_expr.rs:222` (add BoxExpr/ArcExpr handlers)
- Test: `crates/auto-lang/test/a2ts/036_box/box.at`
- Test: `crates/auto-lang/test/a2ts/036_box/box.expected.ts`

**Background:**

The parser at `parser.rs:2516-2532` has hardcoded special handling for `Box` and `Arc`:
```rust
match name.as_str() {
    "Box" => {
        if self.is_kind(TokenKind::LParen) {
            return Ok(Expr::BoxExpr(Box::new(value)));
        }
    }
    "Arc" => { /* similar */ }
}
```

This makes `Box(42)` parse as `Expr::BoxExpr` instead of a normal type construction. In a2ts, `Box` should just be a regular generic type `type Box<T> { value T }`, and `Box(42)` should be `new Box(42)`.

However, the `Box`/`Arc` hardcoding is needed for the Rust backend (a2r). We cannot simply remove it. Two approaches:

**Approach A (recommended): Add BoxExpr/ArcExpr handlers in a2ts**

Add explicit match arms in `ts_expr.rs`:
```rust
Expr::BoxExpr(inner) => {
    out.write(b"new Box(")?;
    self.expr(inner, out)?;
    out.write(b")")
}
Expr::ArcExpr(inner) => {
    out.write(b"new Arc(")?;
    self.expr(inner, out)?;
    out.write(b")")
}
```

This is the minimal change — 4 lines in ts_expr.rs, no parser changes needed.

**Approach B: Make parser backend-aware**

Modify the parser to only emit `BoxExpr`/`ArcExpr` for Rust backend, and treat `Box`/`Arc` as regular type constructors for TS backend. This is more correct but requires threading backend info through the parser, which is a larger change.

**Step 1: Implement Approach A — add handlers in ts_expr.rs**

In `ts_expr.rs`, add two match arms before the `_ =>` fallback:

```rust
// Box/Arc smart pointer construction (from Rust backend hardcoding)
// In TS context, treat as regular constructor calls
Expr::BoxExpr(inner) => {
    out.write(b"new Box(")?;
    self.expr(inner, out)?;
    out.write(b")")
}
Expr::ArcExpr(inner) => {
    out.write(b"new Arc(")?;
    self.expr(inner, out)?;
    out.write(b")")
}
```

## Task 4: Add `Expr::GenName` handler in a2ts

**Files:**
- Modify: `crates/auto-lang/src/trans/ts_expr.rs:222` (add GenName handler)
- Test: `crates/auto-lang/test/a2ts/037_genname/genname.at`
- Test: `crates/auto-lang/test/a2ts/037_genname/genname.expected.ts`

**Background:**

`Expr::GenName` is created when the parser sees a generic name like `Pair<int, str>` in expression position (e.g., as a type argument or factory call). The name is stored as a string like `"Pair<int, str>"`. See `parser.rs:2492-2494`.

In TypeScript, generic names in expression context should map to the base name (TypeScript infers generic params at call sites):
- `Pair<int, str>` → `Pair` (TS infers `<number, string>` from arguments)

**Step 1: Add GenName handler in ts_expr.rs**

```rust
Expr::GenName(name) => {
    // Strip generic args — TypeScript infers them at call sites
    // "Pair<int, str>" → "Pair"
    let base = name.split('<').next().unwrap_or(name);
    out.write_all(base.as_bytes())
}
```

**Step 2: Write the test**

Create `crates/auto-lang/test/a2ts/037_genname/genname.at`:
```auto
fn main() {
    // This exercises the GenName path — generic type used in expression position
    // without a call (e.g., passing as a type-level value)

    print("Pair created")
}

type Pair<T, U> {
    first T
    second U

```

Note: `Pair(1, "hello")` may go through `Expr::Node` (which already works) rather than `Expr::GenName`. The GenName path is hit when the generic name appears without `()` call syntax. If the test doesn't exercise GenName directly, skip this task — it may not be reachable in normal TypeScript output.
